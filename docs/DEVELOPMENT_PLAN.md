# 백테스트K — MVP에서 실서비스까지 개발 기획

작성일: 2026-07-19

> 구현 상태(2026-07-19): M1 정적 데이터마트의 첫 운영형 버전을 연결했다. ETF 147개와 대표지수 3개를 종목별 JSON으로 게시하고, GitHub Actions 일일 갱신·스키마 검증·티커 자동완성을 구현했다. 현재 수정종가 공급자는 `provisional`이며 공공데이터포털 키 기반 원시 시세와 분배금 원장 독립 대사는 상용화 전 후속 과제다.

## 1. 핵심 진단: 병목은 코드가 아니라 데이터

현재 MVP는 계산 엔진·UI·문서가 이미 상당히 잘 되어 있다. TWRR/MWRR 분리, 리밸런싱
회전율 산식, MDD를 단위지수로 계산하는 처리 등은 Portfolio Visualizer와 같은 방향의
올바른 설계다.

실서비스의 성패를 가르는 것은 **"종목코드만 입력하면 진짜 데이터로 결과가 나오는가"**다.
지금은 합성 데모 수익률 + CSV 업로드 방식이라 사용자가 직접 수정주가 CSV를 만들어 와야
하며, 이 진입장벽이 유지되면 제품이 성립하지 않는다. 개발 리소스의 70%를 데이터
파이프라인에, 30%를 앱 개선에 배분한다.

특히 한국 ETF 백테스트에서 **분배금(배당) 미반영은 결과를 크게 왜곡**한다. 월배당 ETF가
인기인 국내 시장에서 분배금 없는 가격 수익률만 쓰면 커버드콜·배당 ETF의 성과가 수 %p씩
과소평가된다. "총수익(TR) 기준"이 이 제품의 핵심 차별점이자 최소 품질 기준이다.

## 2. 단계별 로드맵

### M0 — 지금 있는 것 배포 (1주 이내)

- Vercel 또는 Cloudflare Pages에 배포 + 도메인 연결 (예: backtestk.com)
- 투자 위험 고지·이용약관·개인정보처리방침 페이지 추가
  (README 하단 면책 문구를 정식 페이지로 승격)
- Plausible/GA4 애널리틱스, Sentry 에러 추적 추가
- `git init` + GitHub 리포 생성 (이후 데이터 파이프라인의 기반)

목적: 실사용 피드백 수집 시작. "CSV 업로드가 어렵다"는 불만이 곧 M1의 우선순위 근거가 된다.

### M1 — 실데이터 연결 (4~8주, 최우선)

**서버 없이 시작한다.** 정적 데이터마트 패턴:

```text
공공 API/수집 스크립트 (Python)
  → GitHub Actions 일 1회 cron 실행
  → 종목별 월 수익률 JSON 생성 (예: data/069500.json)
  → 같은 정적 호스팅/CDN에서 서빙
  → 프런트가 fetch로 로드
```

- 대상: 순자산 상위 100~200개 국내 상장 ETF + KOSPI/KOSPI200/S&P500 등 대표 지수
- JSON에 반드시 포함: 월 수익률, 분배금 반영 여부, 데이터 기준일, 출처, 상장일
- 서버·DB 비용 0원, 장애 지점 없음, 기존 정적 앱 구조 그대로 유지
- 앱 쪽 변경: 자산 선택 UI를 데모 8종 → 티커 검색(자동완성)으로 교체

이 단계가 끝나면 "한국판 Portfolio Visualizer"라고 부를 수 있는 최소 상태가 된다.

### M2 — 저장과 공유 (3~4주)

- **공유 링크는 서버 없이 가능**: 포트폴리오 설정을 URL 파라미터(base64)로 인코딩.
  커뮤니티 확산의 핵심 기능이므로 M1 직후 최우선
- 계정/저장은 Supabase(Auth + Postgres 무료 티어): 자체 백엔드 없이 소셜 로그인 + 저장
- 결과 페이지 OG 이미지 생성 (링크 공유 시 미리보기 — 유입에 직접 기여)

### M3 — 한국 특화 (PRODUCT_ROADMAP.md Phase 2)

연금저축·IRP·ISA 계좌별 세금, 환율(달러 노출·환헤지), 인출 시뮬레이션.
세법은 매년 바뀌므로 규칙을 데이터(JSON)로 두고 기준일을 명시하는 버전형 규칙 엔진으로
분리한다.

### M4 — 수익화

PRODUCT_ROADMAP.md의 Free/Plus/Pro 구조를 따르되, 유료화는 **데이터 라이선스 문제가
해결된 뒤**에만 시작한다(5절 참고).

## 3. 데이터 소스 전략

| 데이터 | 소스 | 비고 |
|---|---|---|
| ETF/주식 일별 시세 | 공공데이터포털(data.go.kr) 금융위원회 시세정보 API | 무료·공식. 과거 이력 깊이가 제한적일 수 있어 초기 적재는 별도 확보 필요 |
| ETF 분배금 | SEIBro(예탁결제원), data.go.kr의 예탁결제원 API | 분배락일·지급액으로 수정 수익률 계산 |
| 환율·금리·CPI | 한국은행 ECOS API | 무료·공식·이력 충분. 가장 신뢰할 수 있는 소스 |
| 지수(TR 포함) | KRX 정보데이터시스템 | 웹 다운로드는 자동수집·재배포 제한 유의 |
| 증권사 API | 한국투자증권 KIS Developers 등 | 재배포 제한 — 서비스 데이터 원천으로는 부적합 |

주의점:

1. **pykrx, FinanceDataReader는 내부적으로 KRX/네이버를 스크래핑**한다. 개인 연구용은
   무방하지만 상용 서비스 원천으로는 약관 위반 + 차단 리스크. 프로토타입은 이걸로 빠르게
   만들되, 정식 서비스 전에 공공데이터포털 API 기반으로 교체하는 이중 트랙.
2. **수정주가는 직접 계산**: 원시 종가 + 분배금 + 분할/병합 이력으로 자체 계산하고 산식과
   버전을 저장한다(README 상용화 체크리스트 3번). 이것이 재현 가능한 결과의 토대다.

## 4. 코드 아키텍처 진화

전면 리라이트 불필요. 점진 개선 순서:

1. **계산 엔진 분리 (1순위)**: `runBacktest`, `calculateMetrics`, `monthlyIrr`,
   `maxDrawdownDetails` 등 순수 계산 함수를 DOM 의존성 없는 별도 모듈(`engine/`)로 추출.
   → 단위 테스트 가능, 서버 API 재사용 가능, 골든 데이터 회귀 테스트 가능
2. **테스트 도입**: Vitest + 골든 테스트. 미국 자산 몇 개로 Portfolio Visualizer 결과와
   대조하면 산식 정합성을 외부 기준으로 검증할 수 있다
3. **Vite + TypeScript 마이그레이션**: 기능이 늘기 전에. 프레임워크(React/Svelte)는
   M2 계정 기능 시점에 결정해도 늦지 않다
4. **몬테카를로를 Web Worker로**: 경로 수를 늘려도 UI가 멈추지 않게
5. 수제 canvas 차트는 확장 시 유지보수 비용이 커지므로 lightweight-charts, uPlot 등
   경량 라이브러리 검토

## 5. 법률·규제 체크리스트 (유료화 전 필수)

- **시세 데이터 재배포**: KRX 시세의 상업적 재배포는 원칙적으로 코스콤을 통한 유료 계약
  대상. 공공데이터포털 API는 공공데이터법상 활용이 허용되지만 각 API의 이용조건을 확인하고
  출처를 표시한다. 유료 전환 시점에 데이터 계약 비용을 사업 계획에 반영
- **유사투자자문업**: 백테스트 "도구" 제공은 일반적으로 해당하지 않지만, 개인화된 추천
  형태가 되면 신고 대상이 될 수 있다. 제품 문구를 "분석 도구"로 일관되게 유지하고
  프리셋도 "예시"로 표기
- **면책 고지**: README의 범위 제한 문구를 결과 화면과 PDF/CSV 내보내기에도 포함

## 6. 예상 비용

M0~M2까지 사실상 0원: 정적 호스팅(Vercel/CF Pages 무료), GitHub Actions cron(무료 한도),
Supabase 무료 티어, 도메인 연 2만 원 내외. 비용 발생 시점은 데이터 계약(M4 유료화 전)뿐.

## 7. 지금 당장 할 일 3가지

1. **git init + GitHub 리포 생성 후 현재 상태 그대로 Vercel 배포**
2. **공공데이터포털 금융위 시세정보 API 키 + ECOS API 키 신청** (승인 대기가 있으므로 먼저)
3. **계산 엔진 모듈 분리 + Portfolio Visualizer 대조 테스트 1건** (M1 데이터가 붙기 전에
   산식 신뢰도 확보)
